## 2. Executive Summary

### 2.1 Главный вывод

В проекте уже есть сильный коммерческий фундамент:

- есть сущность тарифного плана с ценой, длительностью, лимитом трафика, лимитом устройств и `features`;
- есть unified checkout, который умеет сочетать базовую цену, партнёрскую наценку, промокод и оплату с кошелька;
- есть инвайты, реферальная комиссия, партнёрские коды, партнёрский заработок и вывод денег;
- есть админские интерфейсы для планов, инвайтов, партнёров и кошелька;
- есть Mini App / frontend / mobile / Telegram bot как каналы продаж и удержания.

### 2.2 Главный продуктовый вывод

Тарифы в CyberVPN можно строить не только по цене и сроку, но и по следующим осям:

- количество устройств;
- лимит трафика;
- набор протоколов;
- доступ к разным классам серверов;
- доступ к отдельным нодам / dedicated-пулам;
- уровень поддержки;
- количество и качество инвайтов;
- участие в loyalty / referral / partner logic;
- доступность trial;
- дополнительные профили конфигурации и сценарии подключения.

### 2.3 Главный технический вывод

Для запуска тарифов типа `Basic / Standard / Pro / Pro+` **не обязательно ломать текущий backend**. Самый безопасный путь:

- оставить внутренние tier-значения `basic / pro / ultra / cyber`;
- использовать их как **внутренние технические категории**;
- а пользовательские названия отдать через `name` и/или `features.display_name`.

То есть:

| Внутренний tier | Витринное название |
|---|---|
| `basic` | Basic |
| `pro` | Standard |
| `ultra` | Pro |
| `cyber` | Pro+ |

Это позволит сохранить текущий enum, API и часть админки.

### 2.4 Главные риски перед коммерческим запуском

- trial описан в проекте противоречиво: где-то `2 дня / 2 ГБ`, где-то `7 дней full access`;
- реализация trial в backend сейчас завязана на `admin_users`, а не на `mobile_users`;
- в коде и документах расходятся реальные платежные каналы;
- в разных документах уже существуют разные примеры тарифной сетки;
- доступ по протоколам и доступ по классам серверов пока не описан как единая каноническая схема в `features`.

## 3. Что уже есть в проекте и влияет на монетизацию

## 3.1 Каналы продукта

По репозиторию видно несколько пользовательских точек входа:

- **Frontend / Mini App** на Next.js: планы, trial, оплата, wallet, referral, partner dashboard.
- **Mobile app** на Flutter: сущности плана, подписки, статусы, устройства, протоколы.
- **Telegram bot**: покупка подписки, конфиги, trial, referral, несколько платёжных каналов.
- **Admin panel**: редактирование планов, инвайтов, партнёров, кошелька.

Практический смысл:

- тарифная модель должна быть едина для всех каналов;
- маркетинговые условия нельзя держать только “в текстах”;
- всё, что влияет на цену и доступ, должно иметь единый источник истины.

## 3.2 Текущая доменная база для тарифов

Факт: в backend есть сущность `SubscriptionPlan` со следующими полями:

- `name`
- `tier`
- `duration_days`
- `traffic_limit_bytes`
- `device_limit`
- `price_usd`
- `price_rub`
- `features`
- `is_active`
- `sort_order`

Следствие:

- тариф в текущей модели это **отдельный SKU**, а не просто “план + период” на лету;
- значит, можно сделать разные условия для `Basic 30 дней`, `Basic 90 дней`, `Basic 365 дней`;
- это особенно важно для инвайтов, скидок и loyalty-логики.

## 3.3 Внутренние уровни тарифов

Факт: в enum уже зафиксированы 4 tier:

- `basic`
- `pro`
- `ultra`
- `cyber`

Следствие:

- именно эти 4 уровня лучше считать текущим техническим пределом без schema changes;
- если бизнес хочет `Basic / Standard / Pro / Pro+`, лучше сначала маппить их на текущие tier;
- если бизнесу принципиально нужны именно `standard` и `pro_plus` на уровне API/БД, это уже отдельная миграция.

## 3.4 Что уже есть для growth и monetization

Факт: в проекте уже присутствуют следующие механики:

- инвайт-коды;
- реферальные коды;
- промокоды;
- партнёрские коды;
- кошелёк;
- вывод средств;
- loyalty-концепция в документации;
- partner dashboard / referral dashboard в UI;
- админские операции для партнёров, инвайтов и кошелька.

Следствие:

- тарифы можно строить не изолированно, а как часть воронки: `trial -> first payment -> invite -> referral -> loyalty -> partner/reseller`.

## 5. Фактическое состояние монетизации в коде

## 5.1 Unified checkout уже поддерживает сложную коммерцию

Факт: логика checkout работает в таком порядке:

1. Базовая цена плана.
2. Плюс партнёрская наценка.
3. Минус промокод.
4. Минус кошелёк.
5. Остаток уходит в gateway.

Что это значит для бизнеса:

- можно одновременно использовать тариф, партнёра, промокод и wallet cashback;
- можно делать “полностью бесплатные” покупки за счёт wallet + promo;
- commission logic считается от **базовой цены**, а не от итоговой discounted price.

Это очень важная точка, потому что она уже задаёт экономику партнёрки и referral.

## 5.2 Инвайты уже завязаны на конкретные SKU

Факт:

- инвайты создаются после оплаты;
- выдача определяется `invite.plan_rules`;
- правило привязано к `plan_id`;
- для инвайта задаются `invite_count`, `free_days`, expiry.

Это сильная основа для тарифной матрицы.

Следствие:

- можно дать 0 инвайтов на месячный базовый план;
- 1 инвайт на полугодовой стандарт;
- 2-3 инвайта на годовой Pro/Pro+;
- отдельные бонусы для annual SKUs не требуют ломать checkout.

## 5.3 Referral уже почти готов к коммерческому запуску

Факт:

- у пользователя есть постоянный referral code;
- referral commission идёт в wallet;
- дефолтная ставка сейчас `10%`;
- поддерживаются режимы:
  - `indefinite`
  - `time_limited`
  - `payment_count`
  - `first_payment_only`

Критичный нюанс:

- `time_limited` в текущей реализации выглядит не до конца доведённым и требует проверки/доделки перед публичным запуском как строгого правила.

## 5.4 Partner model уже поддерживает реселлинг, а не просто “пригласи друга”

Факт:

- клиент может быть привязан к партнёру через partner code;
- у partner code есть `markup_pct`;
- партнёр получает:
  - markup amount;
  - commission amount от base price;
- размер комиссии определяется tier-конфигом по числу клиентов;
- earnings отправляются в wallet.

Это уже не “реферальная программа”, а полноценная **reseller / affiliate-reseller foundation**.

## 5.5 Wallet уже является ключевой частью коммерческой системы

Факт:

- есть баланс;
- есть frozen amount;
- есть история транзакций;
- есть withdrawal flow;
- есть admin top-up;
- есть причины транзакций для referral, partner earning, purchase, withdrawal и т.д.

Практический вывод:

- можно запускать комиссии и партнёрские выплаты без внешней ERP на первом этапе;
- но нужно формализовать payout policy, антифрод и бухгалтерские правила.

## 5.6 Admin уже даёт часть необходимых рычагов

Факт: в admin UI уже есть:

- редактор планов;
- консоль партнёров;
- консоль invite codes;
- консоль wallet ops;
- growth-секции для referral/partners.

Практический вывод:

- коммерческий запуск возможен быстрее, чем если бы этих инструментов не было;
- но не все глобальные monetization-config ключи гарантированно вынесены в удобную UI-настройку.

## 6. Противоречия и риски, которые нужно закрыть до финализации тарифов

## 6.1 Trial сейчас описан по-разному

В проекте одновременно встречаются несколько версий:

- старые docs / bot README: `2 дня, 2 ГБ`;
- frontend Mini App copy: `7 дней, full access`;
- backend use case: `TRIAL_DURATION_DAYS = 7`.

Дополнительный риск:

- backend trial use case сейчас работает через `AdminUserRepository`, а не через `mobile_users`.

Вывод:

- пока у проекта нет одной канонической trial-модели;
- это нужно исправить до запуска тарифов и маркетинга.

## 6.2 Нет единой канонической тарифной сетки

В существующих документах встречаются разные примеры:

- `Basic / Pro / Ultra / Cyber` с одной матрицей лимитов;
- другие значения по устройствам и трафику в menu/docs;
- пользовательский сценарий сейчас уже говорит о `Basic / Standard / Pro / Pro+`.

Вывод:

- сначала нужно утвердить **одну master-модель**;
- потом синхронизировать copy во frontend/mobile/bot/docs.

## 6.3 Плановая модель в разных слоях описана по-разному

Есть несколько представлений планов:

- backend domain: `traffic_limit_bytes`, `device_limit`, `features: dict`;
- Remnawave/API/admin: `dataLimitGb`, `maxDevices`, `features`;
- mobile-слой местами ожидает другую форму ответа.

Вывод:

- нужен единый canonical contract по плану;
- иначе коммерческие условия начнут расходиться между каналами.

## 6.4 Не определён канонический способ кодировать premium-функции плана

Сейчас можно выразить почти любую коммерческую разницу, но нет закреплённого стандарта:

- как хранить доступ к протоколам;
- как хранить server pool;
- как хранить premium nodes;
- как хранить support SLA;
- как хранить eligibility для invite/loyalty/partner perks.

Вывод:

- без typed `features` тарифы будут “разъезжаться” между UI и backend логикой.

## 6.5 Платёжные каналы в проекте не до конца синхронизированы

По коду и docs одновременно фигурируют:

- CryptoBot;
- YooKassa;
- Stripe;
- Telegram Stars;
- wallet;
- местами RevenueCat / store payment narrative.

При этом в реальном checkout-потоке сильнее всего прослеживается CryptoBot + wallet, а в Mini App оплата создаётся как USDT invoice.

Вывод:

- для тарифной стратегии по регионам сначала нужно зафиксировать реальный платежный стек по каждому каналу.

## 6.6 Есть продуктовые рассинхроны по протоколам и delivery format

Примеры:

- mobile code и help-content говорят про VLESS / VMess / Trojan / Shadowsocks и fallback-цепочки;
- SEO/help делают акцент на VLESS Reality;
- Telegram bot README говорит о WireGuard config delivery.

Вывод:

- если тарифы должны различаться по протоколам, сначала надо определить “официальный протокольный каталог” CyberVPN.

## 6.7 Статусы пользователя и trial-state тоже не до конца согласованы

В enum пользовательских статусов фигурируют:

- `active`
- `disabled`
- `limited`
- `expired`

Но в части документации указывается отдельный статус `trial`.

Вывод:

- trial сейчас выглядит как логика поверх аккаунта, а не как полноценный согласованный статус;
- это важно для аналитики, UX и биллинга.

## 7. Рекомендованная коммерческая архитектура тарифов

Ниже не “истина”, а **наиболее безопасная продуктовая схема**, которая хорошо ложится на текущий код.

## 7.1 Рекомендация: оставить 4 внутренних tier, а витринно показывать 4 пользовательских плана

Предлагаемый mapping:

| Internal tier | Display name | Роль |
|---|---|---|
| `basic` | Basic | входной тариф |
| `pro` | Standard | массовый личный тариф |
| `ultra` | Pro | power-user тариф |
| `cyber` | Pro+ | premium / family / dedicated |

Почему это правильно:

- не требует немедленной миграции enum;
- сохраняет совместимость backend/admin/API;
- даёт бизнесу именно те названия, которые он хочет.

## 7.2 Рекомендация: считать период отдельным SKU

Пример SKU-матрицы:

- Basic 30 дней
- Basic 90 дней
- Basic 180 дней
- Basic 365 дней
- Standard 30 дней
- Standard 90 дней
- и так далее

Почему это важно:

- invite rules уже привязаны к `plan_id`;
- проще задавать разные annual perks;
- проще управлять скидками и partner economics;
- проще считать продуктовую аналитику.

## 7.3 Рекомендованная пользовательская матрица тарифов

Ниже пример **рабочей стартовой модели**, которая совпадает с вашим ожиданием “basic / standard / pro / pro+”, но при этом опирается на текущую архитектуру.

| Витринный план | Internal tier | Сегмент | Устройства | Трафик | Серверы | Протоколы | Поддержка | Инвайты | Примечание |
|---|---|---|---|---|---|---|---|---|---|
| Basic | `basic` | solo / entry | 1 | 50-100 ГБ | shared basic | базовый безопасный набор | стандарт | обычно 0, на 12 мес можно 1 | минимальный вход |
| Standard | `pro` | обычный пользователь | 2 | 100-200 ГБ | shared + better pool | расширенный базовый набор | стандарт | 1 invite на 6-12 мес | основной массовый тариф |
| Pro | `ultra` | power user | 5 | 500 ГБ или fair-use unlimited | premium shared | все публичные протоколы | priority | 1-2 invite | тариф “для тех, кому важны протоколы и качество” |
| Pro+ | `cyber` | premium / family / creator | 10 или unlimited | unlimited | premium + dedicated / exclusive | полный набор + premium profiles | VIP | 2-3 invite | отдельные ноды и лучшие условия |

### Комментарий по вашему сценарию

Ваш пример:

- в Basic 1 устройство;
- в Standard 2 устройства;
- в Pro другие протоколы;
- в Pro+ отдельные сервера.

Это **полностью совместимо** с текущей архитектурой, если:

- устройства задавать через `device_limit`;
- протоколы описывать через typed `features`;
- отдельные сервера выражать через `server_pool` / `premium_nodes` / `dedicated_access`.

## 7.4 Рекомендация по периодам и скидкам

Для каждого тарифа логично иметь 4 периода:

- 1 месяц
- 3 месяца
- 6 месяцев
- 12 месяцев

Рабочая коммерческая логика:

- 1 месяц: без бонусов или минимальные бонусы;
- 3 месяца: первый заметный value uplift;
- 6 месяцев: инвайт и/или loyalty boost;
- 12 месяцев: максимальный бонус, годовая скидка, лучший invite package.

## 7.5 Рекомендация по инвайтам как части тарифов

Так как инвайты уже поддерживаются на уровне `plan_id`, их разумно включить в тарифную политику как annual perk.

Пример стартовой политики:

| SKU | Инвайты | Что получает друг | Срок жизни кода |
|---|---|---|---|
| Basic 30d | 0 | — | — |
| Basic 365d | 1 | 7 дней | 30 дней |
| Standard 180d | 1 | 7-14 дней | 30-60 дней |
| Standard 365d | 1 | 14 дней | 60 дней |
| Pro 180d | 1 | 14 дней | 60 дней |
| Pro 365d | 2 | 14-30 дней | 60-90 дней |
| Pro+ 365d | 2-3 | 30 дней | 90 дней |

## 7.6 Рекомендация по loyalty как части тарифа

Loyalty лучше не смешивать с ценой напрямую, а использовать как retention layer:

- Silver: небольшой бонус в днях;
- Gold: priority support;
- Platinum: premium nodes / early access / premium pools.

Это особенно хорошо усиливает Pro и Pro+ без прямого демпинга цен.

## 9. Рекомендованная модель партнёрской программы

## 9.1 Что уже позволяет текущий код

Партнёрка уже может работать как reseller-программа:

- админ назначает пользователя партнёром;
- партнёр создаёт один или несколько кодов;
- код может иметь свою наценку;
- клиент привязывается к партнёру;
- на каждой покупке партнёр получает:
  - наценку;
  - комиссию от базовой цены;
- выплаты копятся в wallet;
- есть withdrawal flow.

Это сильная стартовая база.

## 9.2 Рекомендация: разделить “рефералку” и “партнёрку” по смыслу

### Referral program

Для всех обычных пользователей:

- личный referral code;
- небольшая комиссия;
- без наценки;
- мотивация “пригласи друга”.

### Partner / reseller program

Для одобренных партнёров:

- partner code;
- наценка;
- tiered commission;
- dashboard по заработку;
- вывод средств;
- промо-материалы;
- при необходимости, отдельные тарифные SKU или dedicated pools.

Это разделение уже соответствует текущему коду и снимает путаницу в терминах.

## 9.3 Рекомендованная сегментация партнёров

Можно запускать 3 уровня партнёрки без перепроектирования ядра:

| Уровень | Для кого | Наценка | Комиссия | Комментарий |
|---|---|---|---|---|
| Affiliate | блогеры, обычные рекомендации | 0% | 10-15% | ближе к referral+, без ценовой деформации |
| Reseller | продавцы, TG-каналы, комьюнити-админы | 0-50% | 15-25% | основной партнёрский слой |
| Master Partner | агенты / сети / B2B | 0-100%+ | индивидуально / tiered | может вести dedicated pools и локальную цену |

Важно:

- текущий код допускает наценку до `300%`, но это **технический максимум**, а не обязательно хороший бизнес-дефолт;
- для consumer-friendly рынка лучше зафиксировать более узкие бизнес-лимиты по сегментам.

## 9.4 Рекомендованная экономика партнёрки

Стартовый безопасный вариант:

### Referral

- ставка: `10%`;
- длительность: первые `12 месяцев` или `первые 3-6 оплат`;
- выплата в wallet;
- вывод при минимальном пороге.

### Partner / reseller

- markup: по умолчанию `0-30%` для массовых партнёров;
- комиссия от базы:
  - 0-49 клиентов: `20%`
  - 50-199 клиентов: `25%`
  - 200+ клиентов: `30%`

Это хорошо совпадает с текущей логикой tier config.

## 9.5 Отдельная рекомендация по permanent binding

Сейчас клиент после ввода partner code привязывается к партнёру фактически навсегда.

Это удобно для реселлинга, но рискованно для consumer UX.

Рекомендация:

- сохранить текущую модель для reseller-сценария;
- отдельно решить, нужна ли мягкая политика для B2C:
  - привязка только до первой оплаты;
  - возможность rebinding через support;
  - cooling-off окно.

Если этого не сделать, можно получить спорные кейсы поддержки и злоупотребления.

## 9.6 Что стоит добавить в партнёрскую программу как продуктовую оболочку

Даже если backend уже умеет считать earnings, программе всё ещё нужны продуктовые правила:

- критерии входа в партнёрку;
- правила ручной модерации;
- payout schedule: еженедельно / раз в месяц;
- anti-fraud policy;
- правила self-referral;
- правила chargeback/refund reconciliation;
- пакет маркетинговых материалов;
- лэндинги / deeplink attribution;
- KPI и leaderboard для лучших партнёров.

## 10. Как тарифы, инвайты, referral и partner program должны работать вместе

Рекомендуемая воронка:

1. Пользователь приходит на trial или по invite / referral / partner code.
2. Покупает первый тариф.
3. Получает value от плана:
   - устройства;
   - протоколы;
   - server access;
   - support level.
4. Получает growth-механику:
   - invite bundle;
   - referral code;
   - loyalty progress.
5. Если это активный привлекающий пользователь, он конвертируется:
   - либо в обычного referrer;
   - либо в approved partner.

Правильное разделение ролей:

- **invite** даёт другу бесплатный вход;
- **referral** даёт рекомендателю комиссию;
- **partner** даёт реселлеру право продавать с собственной экономикой;
- **wallet** объединяет всю финансовую мотивацию;
- **loyalty** удерживает клиента внутри экосистемы.

## 12. Что нужно обязательно решить бизнесу перед фиксацией тарифов

Ниже список решений, без которых финальная тарифная сетка будет постоянно меняться.

## 12.1 Канонический trial

Нужно зафиксировать:

- 2 дня или 7 дней;
- есть ли лимит трафика;
- даёт ли trial full access;
- сколько устройств разрешено;
- доступны ли premium/dedicated nodes;
- считается ли trial отдельным статусом.

## 12.2 Каноническая consumer-линейка

Нужно зафиксировать:

- финальные названия планов;
- device limit по каждому плану;
- traffic policy по каждому плану;
- какие протоколы входят;
- какие server pools входят;
- есть ли dedicated access только у top-tier;
- какой support SLA у каждого плана.

## 12.3 Периоды и скидки

Нужно зафиксировать:

- какие периоды продаём;
- размер скидок на 3/6/12 месяцев;
- какие периоды получают инвайты;
- какие периоды сразу бустят loyalty.

## 12.4 Referral policy

Нужно зафиксировать:

- процент;
- длительность attribution;
- allowed/disallowed cases;
- как обрабатываются refund/chargeback.

## 12.5 Partner policy

Нужно зафиксировать:

- кто может стать партнёром;
- какие сегменты партнёров есть;
- допустимая наценка по сегментам;
- комиссия по tier;
- вывод средств и его периодичность;
- что делать с permanent binding.

## 12.6 Payment policy

Нужно зафиксировать:

- какие платёжные методы реально будут в production;
- какие доступны в каком канале;
- нужна ли локальная цена в RUB;
- нужна ли единая прайс-логика между bot, miniapp и web.

- отдельный approval flow;
- разумный business-limit по markup;

- тарифы;
