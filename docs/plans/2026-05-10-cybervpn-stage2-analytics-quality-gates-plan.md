# CyberVPN Stage 2 Analytics And Quality Gates Plan

**Date:** 2026-05-10  
**Scope:** Phase 22 for `10.10.10.34` and the CyberVPN project only.  
**Status:** Implementation plan and operating contract.

---

## 1. Goal

Stage 2 adds broader analytics after S1 is stable. The work must improve business visibility without turning the home server into the only source of production truth.

The Stage 2 goal is to make these areas observable and auditable:

- payments and reconciliation;
- refunds, chargebacks, renewals, and retention;
- subscription expiry and access disablement;
- support SLA and operator action quality;
- public endpoint status data;
- low-impact product analytics;
- log retention and restore evidence;
- release comparison and security quality gates;
- Sentry frontend release health, source maps, and tunnel/proxy routing.

---

## 2. Operating Principles

1. S1 remains the launch baseline. Stage 2 must not block S1 unless a Stage 2 signal exposes an active P0 customer-impacting issue.
2. Metrics must be canonical. Do not create competing definitions for paid conversion, refund rate, chargeback rate, renewal success, or paid-but-no-access.
3. Dashboards must be backed by Prometheus recording rules where the signal is reused in alerts.
4. Product analytics must be asynchronous. Frontend page rendering must not wait on analytics ingestion.
5. PII, tokens, payment payloads, subscription URLs, and support private notes must not be logged or exported to analytics.
6. Release evidence must be append-only. Generated reports should go to evidence storage, not replace prior evidence.
7. Status page data must be derived from synthetic checks and Prometheus, not direct internal service access from a public status page.

## 3. Deliverables

### 3.1 Dashboards

Provision these dashboards under the CyberVPN Grafana folder:

| Dashboard | Purpose | Source file |
|---|---|---|
| Stage 2 Payment Reconciliation | paid-but-no-access, orphan payments, webhook retry/failure, reconciliation age | `infra/grafana/dashboards/stage2-payment-reconciliation-dashboard.json` |
| Stage 2 Refund And Renewal | refunds, chargebacks, renewal success/failure, retention risk | `infra/grafana/dashboards/stage2-refund-renewal-dashboard.json` |
| Stage 2 Subscription Expiry | expiry forecast, grace risk, disable outcomes | `infra/grafana/dashboards/stage2-subscription-expiry-dashboard.json` |
| Stage 2 Support SLA | backlog, overdue SLA, first response, resolution, admin/support actions | `infra/grafana/dashboards/stage2-support-sla-dashboard.json` |
| Stage 2 Status Page Data Source | public endpoint success, latency, TLS expiry, Caddy errors | `infra/grafana/dashboards/stage2-status-page-dashboard.json` |
| Stage 2 Product Analytics | low-impact event ingestion, funnel, Web Vitals, frontend analytics drops | `infra/grafana/dashboards/stage2-product-analytics-dashboard.json` |
| Stage 2 Release Quality Gates | security gates, SBOM freshness, restore drill age, Sentry frontend errors | `infra/grafana/dashboards/stage2-release-quality-dashboard.json` |

The dashboards are generated by:

```bash
python3 scripts/grafana/generate-stage2-dashboards.py
```

### 3.2 Prometheus Rules And Alerts

Stage 2 recording rules and alerts live in:

```text
infra/prometheus/rules/stage2_analytics_alerts.yml
```

Important alert families:

- payment reconciliation backlog;
- refund or chargeback spike;
- renewal failures;
- subscription expiry disable failures;
- support SLA breach;
- public endpoint synthetic failures;
- TLS certificate near expiry;
- analytics ingestion drops;
- overdue restore drill;
- security quality gate failures;
- frontend Sentry error spike.

### 3.4 Status Page Data Source

The status page source must be generated from Prometheus/blackbox data. The status page itself may be hosted outside the home server later.

Initial exporter:

```text
scripts/status/export-status-page-data.sh
```

Output target on the server:

```text
/srv/storage/evidence/status-page/status-page-data.json
```

### 3.5 Release Comparison Reports

Release comparison reports are generated by:

```text
scripts/release/generate-release-comparison-report.sh
```

The script compares two evidence directories or refs and writes a markdown report with:

- changed release manifests;
- SBOM file presence;
- security scan artifact presence;
- changed image digest files;
- missing mandatory evidence;
- operator follow-up checklist.

### 3.6 Sentry Source Maps And Frontend Tunnel

Frontend source maps must be uploaded by CI using:

```text
scripts/sentry/upload-frontend-sourcemaps.sh
```

Rules:

- source maps are uploaded to Sentry but not served publicly;
- release name must match frontend build release;
- CI needs `SENTRY_AUTH_TOKEN`, `SENTRY_ORG`, `SENTRY_PROJECT`, and `SENTRY_URL`;
- tunnel/proxy path must be routed through the public edge to self-hosted Sentry;
- Sentry events must not include secrets, payment payloads, or full subscription URLs.

### 3.7 CI Quality Gates

Stage 2 adds these CI jobs:

- `observability:stage2-artifacts` validates dashboard/rule/target/runbook wiring;
- `quality:release-comparison-report` produces a release comparison evidence report;
- `sentry:frontend-sourcemaps` uploads frontend source maps when explicitly enabled.

Security gates from Phase 20 remain in force:

- gitleaks secret scan;
- npm audit;
- pip audit;
- Trivy and Grype filesystem/container scan;
- SBOM generation.

## 5. Rollout Order

1. Provision Stage 2 dashboards and Prometheus rules.
2. Validate Prometheus rules with `promtool`.
3. Reload Prometheus and let Grafana provisioning pick up dashboards.
4. Add blackbox synthetic targets.
5. Export one status-page data sample.
6. Add monthly scheduled restore drill timer.
7. Add CI jobs and run local artifact validator.
8. Enable Sentry sourcemap upload only after Sentry projects and tokens are created.
9. Wire product analytics ingestion asynchronously.
10. Add missing full Stage 2 metrics in backend/worker as product features stabilize.

---

## 6. Acceptance Criteria

- All Stage 2 dashboard JSON files exist and are valid JSON.
- `stage2_analytics_alerts.yml` is loaded by Prometheus.
- `stage2-public-endpoints.json` contains CyberVPN-only endpoints.
- Stage 2 artifact validator passes.
- Prometheus rule validation passes.
- Grafana has the Stage 2 dashboards provisioned.
- A status-page JSON sample is generated from Prometheus.
- A scheduled restore drill timer is installed or documented as blocked.
- CI includes Stage 2 observability validation, release comparison evidence, and optional Sentry sourcemap upload.
- Phase 22 evidence is saved under `docs/evidence/observability/` and `/srv/storage/evidence/`.

---

## 7. Out Of Scope

- Non-CyberVPN domains.
- Making the home GitLab the only production source of truth.
- Storing production secrets in the repository.
- Publicly serving source maps.
- Blocking S1 launch on Stage 2-only future metric gaps.
