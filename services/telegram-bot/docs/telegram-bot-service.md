## 1. Обзор

### 1.1 Постановка проблемы

CyberVPN использует remnashop (`ghcr.io/snoups/remnashop`) — стороннее open-source решение для Telegram-бота. Это создаёт критические проблемы:

1. **Фрагментация данных** — remnashop хранит данные в собственной PostgreSQL базе, отдельной от CyberVPN Backend. Пользователи, подписки и платежи существуют в двух несинхронизированных хранилищах.
2. **Отсутствие контроля** — невозможно кастомизировать бизнес-логику, добавлять новые платёжные шлюзы или менять UX без форка и поддержки чужого кода.
3. **Дублирование инфраструктуры** — remnashop требует собственный Redis (`remnashop-redis`), увеличивая потребление ресурсов.
4. **Невозможность интеграции** — Backend API уже имеет endpoints для Telegram (`/api/v1/telegram/`), которые remnashop не использует.
5. **Невозможность масштабирования** — собственная модель данных remnashop не позволяет использовать расширения CyberVPN Backend (audit logs, webhook logs, notification queue).

### 1.2 Решение

Микросервис `telegram-bot` на основе aiogram 3, который:
- **Заменяет** remnashop полностью (полный паритет функций + улучшения)
- **Работает через** CyberVPN Backend API (единый источник данных)
- **Использует** общую инфраструктуру (Redis/Valkey 8.1 — один инстанс)
- **Интегрируется** с task-worker для фоновых операций (рассылки, уведомления)
- **Поддерживает** кибернанк-брендинг CyberVPN

### 1.3 Архитектура интеграции

```
┌──────────────────────┐       ┌──────────────────────────┐
│   Telegram Bot API   │◄──────│    Telegram Users         │
│   (Webhook/Polling)  │       │    (subscribers)          │
└──────────┬───────────┘       └──────────────────────────┘
           │
┌──────────▼───────────┐
│   CyberVPN Bot       │
│   (aiogram 3)        │
│   services/          │
│   telegram-bot/      │
└──────────┬───────────┘
           │ httpx (async HTTP)
┌──────────▼───────────┐       ┌──────────────────────────┐
│   CyberVPN Backend   │◄──────│    Redis / Valkey 8.1    │
│   (FastAPI)          │       │    DB=0: backend cache    │
│   /api/v1/*          │       │    DB=1: bot FSM + cache  │
└──────────┬───────────┘       └──────────────────────────┘
           │
┌──────────▼───────────┐       ┌──────────────────────────┐
│   Remnawave Panel    │◄──────│    PostgreSQL 17.7       │
│   (NestJS)           │       │    cybervpn DB           │
│   VPN management     │       └──────────────────────────┘
└──────────────────────┘
```

Бот **не имеет прямого доступа** к PostgreSQL или Remnawave API. Все операции выполняются через CyberVPN Backend API.

## 3. Функциональная декомпозиция

### Capability 1: Регистрация и онбординг пользователей

**Описание:** Обработка первого контакта пользователя с ботом, включая парсинг deep links, проверку условий доступа, регистрацию в системе и приветственное сообщение.

#### Feature 1.1: Обработка команды /start и deep links

**Вход:** Сообщение `/start [payload]` от Telegram
**Выход:** Приветственное сообщение с главным меню

**Поведение:**
- Парсинг payload: `ref_{telegram_id}` (реферал), `promo_{code}` (промокод), закодированный base64
- Если payload содержит реферальную ссылку — сохранение referrer_id
- Если payload содержит промокод — автоактивация после регистрации
- Вызов Backend API для регистрации/получения пользователя
- Определение языка по `message.from_user.language_code`

**Бизнес-правила:**
- Повторный `/start` от существующего пользователя обновляет данные (username, language), не создаёт дубликат
- Deep link payload декодируется через `aiogram.utils.deep_linking.decode_payload` с fallback на raw string
- Реферальная ссылка привязывается только при первой регистрации

#### Feature 1.2: Условия доступа (Access Conditions)

**Вход:** Состояние доступа из конфигурации бота (API)
**Выход:** Экраны принятия правил и подписки на канал

**Поведение:**
- Проверка режима доступа бота: открытый / только приглашённые / обслуживание
- Если включено принятие правил: показ ссылки на правила + кнопка «Принять»
- Если включена проверка канала: кнопка «Перейти в канал» + «Подтвердить»
- Проверка подписки на канал через `bot.get_chat_member(channel_id, user_id)`

**Бизнес-правила:**
- Пользователь не может пользоваться ботом без принятия правил (если включено)
- Пользователь не может пользоваться ботом без подписки на канал (если включено)
- Режим «только приглашённые» допускает только пользователей, пришедших по реферальной ссылке
- Режим обслуживания блокирует всех кроме администраторов

#### Feature 1.3: Регистрация пользователя

**Вход:** Telegram ID, username, language, referrer_id (опционально)
**Выход:** Объект пользователя в Backend

**Поведение:**
- POST к Backend API для создания/обновления пользователя
- Привязка telegram_id к VPN-аккаунту в Remnawave (через Backend)
- Установка языка по умолчанию
- Если есть referrer_id — привязка реферальной связи
- Логирование события `bot.start`

**Бизнес-правила:**
- Один telegram_id = один VPN-аккаунт (уникальность)
- Регистрация может быть отключена глобально через режим доступа
- При повторном /start данные обновляются, но реферальная связь не перезаписывается

### Capability 2: Главное меню и навигация

**Описание:** Отображение главного меню с профилем пользователя, статусом подписки, трафиком и контекстно-зависимыми кнопками.

#### Feature 2.1: Отображение профиля и статуса

**Вход:** telegram_id пользователя
**Выход:** Форматированное сообщение с профилем

**Поведение:**
- Запрос к Backend API: GET `/api/v1/telegram/user/{telegram_id}`
- Отображение:
  - Профиль: ID, имя, username
  - Статус подписки: ACTIVE / EXPIRED / LIMITED / DISABLED / NONE
  - При ACTIVE: план, трафик (использовано/лимит), время до истечения, устройства
  - При EXPIRED: сообщение об истечении (разное для trial и платной подписки)
  - При LIMITED: сообщение о трафике (с информацией о стратегии сброса: NO_RESET/DAY/WEEK/MONTH)
  - При DISABLED: сообщение о блокировке + контакт поддержки
  - При NONE: предложение trial (если доступен) или покупки

**Формат сообщения:**
```
👤 Профиль
<blockquote>
• ID: 123456789
• Имя: Иван
• Username: @ivan
</blockquote>

💳 Подписка
<blockquote>
• План: CyberPro (Pro)
• Трафик: 45.2 / 100 ГБ
• Осталось: 23 дня
• Устройства: 2 / 5
</blockquote>
```

#### Feature 2.2: Контекстно-зависимые кнопки

**Вход:** Статус подписки пользователя
**Выход:** Reply keyboard с кнопками

**Поведение:**
- ACTIVE: «🚀 Подключиться», «📱 Мои устройства», «💳 Подписка», «👥 Пригласить», «🆘 Поддержка»
- EXPIRED/LIMITED: «⚠️ [статус]», «💳 Подписка», «👥 Пригласить», «🆘 Поддержка»
- NONE + trial доступен: «🎁 ПОПРОБОВАТЬ БЕСПЛАТНО», «💳 Подписка», «👥 Пригласить», «🆘 Поддержка»
- NONE + trial недоступен: «💳 Подписка», «👥 Пригласить», «🆘 Поддержка»
- DISABLED: «🆘 Поддержка»
- Если пользователь — администратор: дополнительная кнопка «🛠 Панель управления»

### Capability 3: Управление подпиской

**Описание:** Полный цикл управления подпиской: просмотр, покупка, продление, смена плана. Реализован как FSM (Finite State Machine) с состояниями.

#### Feature 3.1: Просмотр текущей подписки

**Вход:** Нажатие кнопки «💳 Подписка» или callback `subscribe`
**Выход:** Экран подписки с вариантами действий

**Поведение:**
- Если подписка активна: показ деталей + кнопки «Продлить» / «Изменить»
- Если подписка истекла: показ деталей + кнопка «Продлить» / «Новая подписка»
- Если нет подписки: кнопка «Купить подписку» + «Активировать промокод»

#### Feature 3.2: Выбор плана

**Вход:** Нажатие «Купить подписку» / «Продлить» / «Изменить»
**Выход:** Inline-клавиатура с планами
**FSM-состояние:** `SubscriptionStates.selecting_plan`

**Поведение:**
- Запрос доступных планов из Backend API
- Фильтрация по доступности (ALL / NEW / EXISTING / INVITED / ALLOWED)
- Отображение для каждого плана:
  - Название и описание
  - Лимит трафика (или «безлимит»)
  - Лимит устройств (или «безлимит»)
  - Диапазон цен (от минимальной длительности)
- Статус плана (🟢/🔴)

**Бизнес-правила:**
- Планы типа TRIAL не показываются в списке покупки
- Планы с доступностью ALLOWED показываются только разрешённым пользователям
- Планы с доступностью NEW показываются только пользователям без истории подписок
- Планы с доступностью INVITED показываются только пришедшим по реферальной ссылке

#### Feature 3.3: Выбор длительности

**Вход:** Выбор плана (callback `select_plan:{plan_id}`)
**Выход:** Inline-клавиатура с длительностями
**FSM-состояние:** `SubscriptionStates.selecting_duration`

**Поведение:**
- Запрос длительностей для выбранного плана
- Отображение каждой длительности с ценой:
  - `⌛ 30 дней | 299₽`
  - `⌛ 90 дней | 699₽`
  - `⌛ 180 дней | 1199₽`
  - `⌛ 365 дней | 1999₽`
  - `⌛ Безлимит | 🎁` (бесплатные планы)
- Применение скидок:
  - Персональная скидка пользователя
  - Скидка на следующую покупку (одноразовая)
  - Промокод (если активирован в этой сессии)

**Бизнес-правила:**
- Цена отображается в валюте по умолчанию для шлюза
- Если скидка > 0%, показывается зачёркнутая старая цена и новая
- Длительность -1 означает «безлимит» (навсегда)

#### Feature 3.4: Выбор способа оплаты

**Вход:** Выбор длительности (callback `select_duration:{duration_id}`)
**Выход:** Inline-клавиатура со шлюзами
**FSM-состояние:** `SubscriptionStates.selecting_payment`

**Поведение:**
- Список включённых платёжных шлюзов:
  - CryptoBot — криптовалюта (USDT, TON)
  - YooKassa — банковские карты (RUB)
  - Telegram Stars — внутренняя валюта Telegram (XTR)
- Цена конвертируется в валюту каждого шлюза
- Формат: `{gateway_icon} {name} | {price} {currency}`

**Бизнес-правила:**
- Шлюзы отображаются в порядке, заданном администратором (placement)
- Неактивные шлюзы не отображаются
- Если доступен только 1 шлюз — пропуск этого шага

#### Feature 3.5: Подтверждение покупки

**Вход:** Выбор способа оплаты
**Выход:** Сводка заказа + кнопка «Оплатить»
**FSM-состояние:** `SubscriptionStates.confirming`

**Поведение:**
- Отображение полной сводки:
  - План, длительность, способ оплаты
  - Финальная сумма
  - Тип операции: ПОКУПКА / ПРОДЛЕНИЕ / ИЗМЕНЕНИЕ
- При продлении: «⚠️ Текущая подписка будет продлена на выбранный срок»
- При изменении: «⚠️ Текущая подписка будет заменена без пересчёта остатка»
- Кнопки: «💳 Оплатить», «⬅️ Изменить способ оплаты», «⬅️ Изменить длительность»

### Capability 4: Обработка платежей

**Описание:** Три платёжных шлюза с единым интерфейсом: создание инвойса, отслеживание статуса, обработка подтверждения.

#### Feature 4.1: Оплата через CryptoBot

**Вход:** Подтверждение покупки с выбранным шлюзом CryptoBot
**Выход:** Inline-кнопка с ссылкой на оплату

**Поведение:**
- Создание инвойса: POST `/api/v1/payments/crypto/invoice`
  - amount, currency (USDT/TON), user_uuid, plan_id
- Отправка web_app кнопки с URL инвойса
- Webhook от CryptoBot → Backend → уведомление бота
- Timeout: 24 часа (отмена инвойса)

**Бизнес-правила:**
- Инвойс содержит payload: `{plan_id}:{telegram_id}:{duration_days}`
- Подпись webhook верифицируется через HMAC-SHA256(SHA256(token))
- При истечении timeout — отмена и уведомление пользователя

#### Feature 4.2: Оплата через YooKassa

**Вход:** Подтверждение покупки с выбранным шлюзом YooKassa
**Выход:** Inline-кнопка с ссылкой на оплату

**Поведение:**
- Создание платежа через YooKassa API
  - amount (RUB), description, confirmation (redirect URL)
  - metadata: telegram_id, plan_id, duration_days
- Отправка web_app кнопки с confirmation.confirmation_url
- Webhook от YooKassa → Backend → уведомление бота
- Timeout: 30 минут

**Бизнес-правила:**
- Тестовый режим через `YOOKASSA_TEST_MODE` для отладки
- Подпись webhook через IP whitelist + secret

#### Feature 4.3: Оплата через Telegram Stars

**Вход:** Подтверждение покупки с выбранным шлюзом Stars
**Выход:** Inline-инвойс Telegram

**Поведение:**
- Отправка инвойса через `bot.send_invoice()`:
  - `provider_token=""` (пустой для Stars)
  - `currency="XTR"` (Telegram Stars)
  - `prices=[LabeledPrice(label, amount_in_stars)]`
  - `payload="{plan_id}:{telegram_id}:{duration_days}"`
- Обработка `PreCheckoutQuery` — ответ `ok=True` в течение 10 секунд
- Обработка `SuccessfulPayment`:
  - Сохранение `telegram_payment_charge_id` (для возвратов)
  - Активация подписки через Backend API

**Бизнес-правила:**
- Цены в Stars: месяц=100, квартал=250, год=800 (настраиваемо)
- `PreCheckoutQuery` ОБЯЗАТЕЛЕН — Telegram ждёт ответ ≤10 секунд
- `telegram_payment_charge_id` необходим для refund через Telegram

## 8. Архитектурные решения

<architecture>

### Решение 1: aiogram 3 вместо python-telegram-bot

**Обоснование:**
- aiogram 3 — async-first (asyncio native), что соответствует стеку проекта (FastAPI, httpx, SQLAlchemy async)
- Встроенная поддержка FSM с Redis storage
- Middleware system аналогичен FastAPI middleware
- De-facto стандарт для русскоязычного Telegram-бот разработки
- Поддержка Fluent i18n (aiogram-i18n пакет)
- Активная разработка и сообщество

**Альтернативы отклонены:**
- `python-telegram-bot` — sync-first, менее гибкая middleware система
- `Telethon` — MTProto (overkill для Bot API)
- `grammY` (TypeScript) — не Python, не совместим с Python SDK

### Решение 2: Backend API как единственный источник данных

**Обоснование:**
- Единый источник истины (single source of truth)
- Бот не требует собственной базы данных
- Все изменения проходят через Backend → аудит, валидация, бизнес-логика
- Позволяет заменить бота без потери данных
- Backend уже имеет endpoints: `/api/v1/telegram/`, `/api/v1/payments/`, `/api/v1/users/`

**Компромиссы:**
- Дополнительный HTTP hop на каждый запрос (+10-50ms)
- Зависимость от доступности Backend API
- Необходимо расширить Backend API для недостающих admin endpoints

### Решение 3: Webhook mode (production) + Polling mode (development)

**Обоснование:**
- Webhook: эффективнее (нет polling overhead), обязателен для Telegram Stars
- Polling: проще для локальной разработки (не нужен HTTPS endpoint)
- Переключение через `BOT_MODE` env var

### Решение 4: Fluent (Project Fluent) для i18n

**Обоснование:**
- remnashop уже использует Fluent (.ftl файлы) — миграция переводов
- Сложные правила плюрализации для русского языка (one/few/many/other)
- Pattern matching лучше gettext для параметризованных строк
- `fluent.runtime` Python библиотека

### Решение 5: Общий Redis с backend и task-worker

**Обоснование:**
- Устраняет remnashop-redis контейнер
- Изоляция через Redis DB numbers: DB=0 (backend/worker), DB=1 (bot FSM + cache)
- Valkey 8.1 поддерживает достаточно соединений

### Решение 6: Расширение Backend API для admin функций

**Обоснование:**
- Admin функции бота (статистика, рассылки, планы, промокоды) требуют endpoints, которых может не хватать в текущем Backend API
- Вместо прямых вызовов к БД — расширение Backend API новыми endpoints
- Это обеспечивает единообразие и переиспользование (admin panel Next.js может использовать те же endpoints)

</architecture>

## 11. Риски и митигации

<risks>

| # | Риск | Влияние | Вероятность | Митигация |
|---|------|---------|-------------|-----------|
| 1 | **Telegram Bot API rate limits** (30 msg/sec для рассылок) | Высокое | Высокая | Реализовать queue с rate limiting. Использовать task-worker для фоновых рассылок |
| 2 | **Backend API недоступен** | Критическое | Средняя | Circuit breaker pattern. Graceful degradation: показ кэшированных данных. Retry с exponential backoff |
| 3 | **Stars payment timeout** (PreCheckoutQuery ≤10 сек) | Высокое | Средняя | Минимизировать логику в pre_checkout handler. Только OK response, активация async |
| 4 | **Недостаточно Backend API endpoints** для admin функций | Среднее | Высокая | Планировать расширение Backend API параллельно с разработкой бота. API-first подход |
| 5 | **Миграция пользователей с remnashop** | Высокое | Средняя | Импорт из remnashop DB. Параллельная работа обоих ботов в переходный период |
| 6 | **Fluent i18n complexity** (русская плюрализация) | Среднее | Средняя | Копировать проверенные .ftl из remnashop. Extensive testing плюрализации |
| 7 | **aiogram 3 breaking changes** | Среднее | Низкая | Pinning версии в pyproject.toml. Мониторинг changelog |
| 8 | **CryptoBot/YooKassa webhook delivery failures** | Высокое | Низкая | Webhook retry логика (exponential backoff). Ручная проверка статуса через API |
| 9 | **Redis memory overflow** (FSM state accumulation) | Среднее | Низкая | TTL на FSM state (1 час). Мониторинг Redis memory. Очистка стейтов по cron |
| 10 | **Scope creep** (полный паритет с remnashop = большой объём) | Высокое | Высокая | Фазовая реализация. MVP = Фазы 0-5 (user flow). Admin = Фаза 8 (отдельный milestone) |

</risks>

## 12. Ограничения и допущения

<constraints>

### Ограничения

1. **Бот не имеет прямого доступа к БД** — все операции через CyberVPN Backend API
2. **Один Telegram bot token** — один инстанс бота (без горизонтального масштабирования Dispatcher)
3. **Webhook mode требует HTTPS** — необходим SSL-сертификат и публичный домен
4. **Telegram Stars refund** — возврат возможен только с сохранённым `telegram_payment_charge_id`
5. **Fluent .ftl формат** — переводчики должны знать синтаксис Fluent
6. **Redis обязателен** — FSM storage требует Redis (не поддерживается in-memory в production)

### Допущения

1. **Backend API доступен и стабилен** — SLA >99.5%
2. **Redis/Valkey доступен** — единый инстанс для всех сервисов
3. **Backend API будет расширен** для недостающих admin endpoints (статистика, рассылки, промокоды, планы)
4. **Remnawave API стабилен** — Backend корректно проксирует все вызовы
5. **Один разработчик** — фазовая реализация, не параллельная
6. **Пользователи remnashop будут мигрированы** через импорт/синхронизацию
7. **Цены и планы управляются через admin dashboard** (бот или Next.js), не через env vars

</constraints>
